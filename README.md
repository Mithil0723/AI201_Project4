# Provenance Guard

A backend service that any creative-sharing platform can plug in to answer one
question about a piece of text: *was this written by the person who posted it, or
generated by AI and passed off as theirs?* It classifies submitted text with a
three-signal ensemble, scores its confidence honestly (it's allowed to say "I
don't know"), shows the reader a plain-language label, lets creators appeal a
call they disagree with, and keeps a structured audit log behind rate limiting.

The goal isn't to police creativity. It's attribution and trust: giving readers
context and giving creators a fair process.

---

## Quick start

```bash
python -m venv .venv
source .venv/bin/activate          # Windows: .venv\Scripts\activate
pip install -r requirements.txt

# create .env with your Groq key (already gitignored)
echo "GROQ_API_KEY=your_key_here" > .env

python app.py                      # uses the real Groq LLM signal
# or, to run with no network/key (offline test mode):
MOCK_LLM=1 python app.py
```

Server runs on `http://localhost:5000`.

| Endpoint        | Method | Body / params                          | Purpose                         |
|-----------------|--------|----------------------------------------|---------------------------------|
| `/submit`       | POST   | `{ text, creator_id }`                  | classify + label a submission   |
| `/appeal`       | POST   | `{ content_id, creator_reasoning }`    | contest a classification        |
| `/log`          | GET    | `?limit=N`                             | recent audit entries (JSON)     |
| `/health`       | GET    | —                                      | liveness check                  |

> **A note on the LLM signal.** Signal 1 calls Groq's `llama-3.3-70b-versatile`.
> When you run the app on your own machine with a key, it makes a real call. The
> code also ships an offline mode (`MOCK_LLM=1`) that substitutes a deterministic
> stub derived from the two structural signals, so the full ensemble can be
> exercised — and graded — without network access or burning API calls. Every
> number shown in this README was produced in that offline mode; with the live
> model the LLM score replaces the stub and the same pipeline runs unchanged.

## Architecture overview — the path a submission takes

A `POST /submit` with `{ text, creator_id }` arrives. The raw text fans out to
three independent detectors (`signals.py`): the Groq LLM judges it semantically,
the stylometric heuristic measures how uniform the prose is, and the lexical
detector counts AI-favoured vocabulary. Each returns a score in 0–1 on the same
"how likely is this AI" axis. `run_ensemble` combines them with a weighted
average into a single **confidence**. That confidence goes to `make_label`
(`labels.py`), which picks one of three attributions and writes the exact label
text the reader sees. The submission and a full audit row are written to SQLite,
and the JSON response comes back with the `content_id`, attribution, confidence,
label, and every per-signal score. A diagram of both flows is in
[`planning.md`](./planning.md#architecture).

## Detection signals — what each measures, why, and what it misses

Three signals, each looking at a genuinely different property of the text. The
third is the **ensemble stretch feature** (see below).

**1. LLM semantic judgement — Groq `llama-3.3-70b-versatile`.**
*Measures:* whether the writing reads as human or machine — idea development,
voice, specific lived detail vs. evenly-hedged generic prose. The model is asked
for a JSON `ai_likelihood` (0–1) at temperature 0 for stability.
*Why:* this is the only signal that understands *meaning*; it catches fluent AI
that's statistically unremarkable.
*Misses:* lightly human-edited AI, and it can be over-eager to call polished
human writing "AI."

**2. Stylometric heuristics — pure Python.**
*Measures:* how *uniform* the prose is, via sentence-length burstiness
(coefficient of variation), type-token ratio, and punctuation variety. AI text
is smooth and even; humans are burstier and use more varied punctuation.
*Why:* it's structural, not semantic — it can flag machine-like evenness the LLM
might rate as "well written." Cheap, deterministic, no API.
*Misses:* short texts (too few sentences to measure variance) and naturally
formal, tidy human writers.

**3. Lexical AI-marker density — pure Python.**
*Measures:* per-100-word density of words/phrases LLMs overuse ("furthermore,"
"delve," "it is important to note," "leverage," "paradigm," "tapestry"…).
*Why:* word choice is a third, independent axis; it catches the telltale
"AI register" even when structure looks fine.
*Misses:* genuinely academic humans trip it; an AI told to "write casually"
dodges it.

These are **semantic / structural / lexical** — three different failure modes, so
the combination is more informative than any one. Where they disagree, the
combined score lands near 0.5, which is the honest answer.

**How they're combined.** Weighted average — LLM `0.50`, stylometry `0.30`,
lexical `0.20`. The LLM leads because semantic judgement is the strongest single
signal; the two structural signals are an independent cross-check that can pull a
confident LLM call back toward "uncertain." If the LLM is unavailable, its weight
is redistributed (stylometry `0.60`, lexical `0.40`, re-normalised) so a missing
signal never silently drags confidence to zero.

## Confidence scoring — what it means and how I validated it

`confidence` is the system's estimated probability the text is **AI-generated**:
`0.0` = confidently human, `0.5` = genuinely undecided, `1.0` = confidently AI.
I picked what 0.5 should mean to a *user* first — "we don't know" — then built the
scoring to land there when signals disagree, rather than forcing a binary flip at
0.5.

The thresholds are deliberately asymmetric, because on a writing platform
**wrongly flagging a real person as AI is the worse error**:

| confidence    | attribution    |
|---------------|----------------|
| `< 0.40`      | `likely_human` |
| `0.40 – 0.70` | `uncertain`    |
| `> 0.70`      | `likely_ai`    |

The "likely AI" bar sits high and the uncertain band is wide so the system says
"inconclusive" rather than accusing.

**How I validated it's meaningful.** I ran the four calibration inputs from the
spec plus a deliberately heavy-AI sample and checked the scores matched intuition
and that all three label bands are reachable. Results (offline mode):

| input                         | confidence | attribution    |
|-------------------------------|-----------:|----------------|
| clearly human (ramen review)  | **0.152**  | likely_human   |
| lightly-edited AI (remote work) | 0.136    | likely_human   |
| formal human (monetary policy)| 0.383      | likely_human   |
| clearly AI (paradigm shift)   | 0.671      | uncertain      |
| heavy AI (marker-stuffed)     | **0.769**  | likely_ai      |

The clearly-AI sample landing at 0.671 *uncertain* rather than *likely_ai* is the
asymmetry working as designed: only the lexical signal was fully confident, and
one signal isn't allowed to clear a deliberately high bar on its own. With the
live Groq model the semantic score pushes that case firmly into `likely_ai`.

### Two example submissions with noticeably different confidence

**Low-confidence-of-AI (high confidence it's human) — `0.152`:**
> ✍️ Likely human-written. Our automated analysis found no strong signs of AI
> generation (estimated 85% human-written). This is an estimate, not a guarantee.

Signal scores: `llm=0.152, stylometry=0.254, lexical=0.0` — all three agree.

**High-confidence-of-AI — `0.769`:**
> 🤖 Likely AI-generated. Our automated analysis estimates a 77% chance this text
> was produced with generative AI. This is an estimate, not a verdict — the
> creator can appeal if they believe it's wrong.

Signal scores: `llm=0.769, stylometry=0.615, lexical=1.0` — all three lean AI.

The ~0.62 spread between these two confirms the scoring produces real variation,
not a constant near 0.5.

## Transparency label — all three variants

The label changes with the confidence score. Exact text (the percentages are
filled from the confidence at runtime):

| Variant | Trigger | Exact label text |
|---|---|---|
| **High-confidence AI** | `confidence > 0.70` | `🤖 Likely AI-generated. Our automated analysis estimates a {ai_pct}% chance this text was produced with generative AI. This is an estimate, not a verdict — the creator can appeal if they believe it's wrong.` |
| **High-confidence human** | `confidence < 0.40` | `✍️ Likely human-written. Our automated analysis found no strong signs of AI generation (estimated {human_pct}% human-written). This is an estimate, not a guarantee.` |
| **Uncertain** | `0.40 ≤ confidence ≤ 0.70` | `❓ Uncertain origin. Our analysis couldn't confidently tell whether a human or AI wrote this (roughly {ai_pct}% toward AI, {human_pct}% toward human). We're flagging it as inconclusive rather than guessing.` |

Every variant calls itself an estimate, and the AI/uncertain ones point to the
appeal path.

## Appeals workflow

A creator who disagrees with a classification sends their `content_id` and a
free-text `creator_reasoning` to `POST /appeal`. The system:

1. looks up the submission (returns `404` if the `content_id` is unknown),
2. flips its status from `classified` to `under_review`,
3. writes an `audit_log` row (`event="appeal"`) carrying the creator's reasoning
   *and* a copy of the original attribution and signal scores, so the decision
   and the challenge sit side by side, and
4. returns a confirmation echoing the original decision back.

Automated re-classification is intentionally not done — an appeal is a request
for *human* review. A reviewer opens the queue by reading the audit rows where
`status="under_review"`.

```bash
curl -s -X POST http://localhost:5000/appeal -H "Content-Type: application/json" \
  -d '{"content_id":"PASTE-ID","creator_reasoning":"I wrote this myself; English is my second language so it reads informally."}'
```
```json
{
  "content_id": "514983df-...",
  "status": "under_review",
  "message": "Appeal received. This classification is now under human review.",
  "original_decision": { "attribution": "likely_human", "confidence": 0.152 },
  "appeal_reasoning": "I wrote this myself; English is my second language..."
}
```

## Rate limiting

Applied to `/submit` with Flask-Limiter (in-memory store):

```python
@limiter.limit("10 per minute;100 per day")
```

**Why these numbers.** A real creator submits their own work a handful of times
an hour at most — even an intense edit-and-resubmit session rarely exceeds a few
calls a minute, so **10/minute** leaves comfortable headroom for honest use while
a script trying to flood or fuzz the classifier hits the wall almost immediately.
**100/day** caps sustained automated abuse (and runaway Groq spend) without ever
getting in a busy human's way. The limit is per-IP; a production deployment would
key it per authenticated creator account instead.

**Evidence** — 12 rapid requests, first 10 pass, rest are rejected:
```
200 200 200 200 200 200 200 200 200 200 429 429
```
A rejected request returns a structured body:
```json
{ "error": "rate_limit_exceeded", "detail": "10 per 1 minute",
  "message": "Too many submissions. Please slow down and try again shortly." }
```

## Stretch feature — Ensemble detection (3+ signals)

Built into the core pipeline rather than bolted on. Three signals (LLM,
stylometry, lexical) run on every submission and combine through a documented
**weighted scheme** — `0.50 / 0.30 / 0.20`, re-normalised to `0.60 / 0.40` when
the LLM is unavailable. Alongside the weighted score, the response reports a
transparency-only **majority vote** (`"2/3 signals lean AI"`) so a reader can see
how much the detectors agreed, though the vote does not drive the decision. The
weighting rationale is in the *Detection signals* section above and in
`planning.md`.

## Known limitations

The clearest failure case is **formal, tidy human writing** — an academic
abstract or a careful policy paragraph. Its even sentence lengths defeat the
stylometry burstiness metric and its formal vocabulary trips the lexical detector,
so two of three signals push toward AI even though a person wrote it. The system
survives the spec's monetary-policy sample (it lands at 0.383, `likely_human`),
but a denser formal passage would cross into `uncertain` and get hedged when it
shouldn't. This is intrinsic to the signals: "uniform and formal" is exactly what
two of my three detectors treat as AI-like, and only the LLM can really tell a
careful human apart from a machine there. A second weak spot is **very short
text** (a haiku, a caption): with one or two sentences there's not enough material
to measure variance or diversity, so the structural signals go blind and the
honest output is "uncertain."

## Spec reflection

**Where the spec helped.** Writing the *Uncertainty representation* section
before any code forced me to decide what 0.5 means to a user *first*. That single
decision drove everything downstream — the asymmetric thresholds, the wide
uncertain band, the "this is an estimate" wording in every label. If I'd coded
first I'd have built a 0.5 binary flip and discovered the problem only when trying
to explain a 0.62 to someone.

**Where it diverged.** My plan treated the third signal as a stretch to add after
the two-signal core worked. In practice it was cheaper to write all three pure-ish
functions against one shared interface from the start, so the ensemble was
three-wide from the first run rather than retrofitted. I also added the
LLM-unavailable re-weighting path, which the spec didn't call for — it became
necessary the moment I needed the system to run and be tested without live Groq
access.

## AI usage

**1. Generating the stylometry signal.** I gave the AI tool the *Detection
signals* section and asked for `signal_stylometry` computing sentence-length
variance, type-token ratio, and punctuation variety, each normalised to 0–1. The
first version returned raw TTR directly as the score, which made short human texts
(high TTR) read as AI — backwards. I overrode it: length-corrected the TTR,
damped its influence under 40 words, and made burstiness the dominant sub-metric,
since uniform sentence length is the more reliable structural tell.

**2. The label-mapping function.** I asked for `make_label(confidence)` returning
the three variants from my spec. The generated version used symmetric cutoffs
(0.33 / 0.66) and flipped straight from "human" to "AI." I revised it to the
asymmetric 0.40 / 0.70 thresholds and rewrote every label string to call itself
an *estimate* and surface the appeal path — the spec's whole point was that a
false "AI" flag is the costly error, and the AI's symmetric default ignored that.
